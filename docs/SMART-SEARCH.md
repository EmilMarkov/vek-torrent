# Умный поиск: анализ (пункт 10)

Документ-рассуждение о том, можно ли сделать поиск умнее стандартного поиска
rutracker. Это анализ вариантов и рекомендация, а не описание реализации.

## Что есть сейчас

Поиск идёт через `tracker.php` rutracker: серверный полнотекстовый поиск по
заголовкам раздач с сортировкой и пагинацией. Поверх мы уже добавили:

- **lazy-search** — запрос уходит с задержкой, устаревшие ответы отменяются;
- **мгновенные клиентские фильтры** по уже загруженной выдаче (текст с
  исключениями `-слово`, размер, сиды, раздел, статус проверки, сортировка).

Ограничения родного поиска rutracker: ищет в основном по названию темы,
чувствителен к формулировке (порядок слов, регистр, латиница/кириллица,
пунктуация), не понимает синонимов, опечаток и намерения («сериал X в 4K с
Atmos»).

## Куда можно расти (по возрастанию сложности)

### 1. Нормализация и расширение запроса (дёшево, локально)

Перед отправкой запроса на трекер:

- приводить регистр, убирать лишнюю пунктуацию, нормализовать пробелы;
- разворачивать частые сокращения и синонимы (4K↔2160p, Atmos↔E-AC3 JOC,
  «сезон»↔«season»↔«S01»);
- транслитерация RU↔EN и подсказки раскладки («заменить неверную раскладку»),
  чтобы «����» превращалось в осмысленный запрос;
- строить несколько вариантов запроса и объединять результаты (дедуп по
  `topic_id`).

Плюсы: не требует внешних сервисов, работает офлайн, предсказуемо. Минусы:
словари синонимов нужно вести вручную.

### 2. Устойчивость к опечаткам и ранжирование выдачи (средне, локально)

- нечёткое сопоставление (расстояние Левенштейна / триграммы) уже на клиенте
  для подсветки и переранжирования результатов;
- собственный скоринг релевантности: комбинировать текстовую близость,
  количество сидов, статус проверки, свежесть, соответствие раздела запросу —
  и сортировать по нему вместо «как отдал трекер»;
- «умные подсказки»: если у запроса мало результатов — предлагать ослабить
  (убрать год/качество) либо исправить опечатку.

Плюсы: заметный прирост качества без внешних зависимостей. Минусы: скоринг надо
калибровать, чтобы не «перемешивать» очевидно релевантное.

### 3. Обогащение метаданными (средне, внешние API)

Многие раздачи — фильмы/сериалы/игры. Можно подтягивать метаданные из внешних
каталогов (напр. кинобазы) для:

- нормализации названий (оригинальное + локализованное, год, альтернативные
  имена) и автоматического расширения запроса;
- фасетов «жанр/год/студия», отсутствующих в rutracker.

Плюсы: качественный скачок для медиа. Минусы: зависимость от сторонних API,
лимиты, приватность, необходимость мэтчинга «раздача ↔ произведение».

### 4. Семантический поиск (дорого)

Локальные эмбеддинги названий раздач и поиск по смыслу (векторная близость)
поверх периодически обновляемого индекса категорий/популярных раздач. Позволяет
искать по описанию намерения, а не по точным словам.

Плюсы: самый «умный» вариант. Минусы: нужен индекс и его обновление, вес модели,
ощутимая сложность; для трекера с миллионами тем полный индекс нереалистичен —
разумнее индексировать локально то, что пользователь уже видел/скачивал.

## Рекомендация

Оптимальное соотношение польза/сложность — **этапы 1 и 2**: нормализация и
расширение запроса плюс собственный скоринг релевантности и устойчивость к
опечаткам. Это чисто локальные, приватные и предсказуемые улучшения, которые
хорошо ложатся на уже имеющуюся архитектуру (расширение запроса — в крейте
`rutracker`/`vek-core`, скоринг и подсказки — рядом с текущими клиентскими
фильтрами). Этапы 3–4 стоит рассматривать позже и только для медиа-раздач, взвесив
зависимость от внешних сервисов.
